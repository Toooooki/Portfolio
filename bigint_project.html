Overview / プロジェクト概要

C言語の標準型（long long等）では扱いきれない巨大な桁数の整数を計算するためのライブラリをゼロから実装しました。 四則演算に加え、ユークリッドの互除法を用いた最大公約数（GCD）の算出、それを利用した分数の約分機能なども搭載しています。

既存のライブラリ（GMP等）を使用せず、配列とポインタ操作によって筆算のロジックをコード化することで、 コンピュータが数値をどうメモリ上で扱い、計算しているのかという基礎原理の理解を深めることを目的として開発しました。

Technical Highlights / 技術的なこだわり

構造体によるデータ抽象化

数値の符号と各桁の数値を構造体としてパッケージ化し、ポインタ渡しで関数間を連携させる設計にしました。 また、整数だけでなく分数構造体（BUNSU）を定義し、分母・分子をそれぞれ多倍長整数で保持することで、誤差のない有理数計算を実現しました。

struct NUMBER { int n[KETA]; // 各桁の値を格納する配列 int sign; // 符号 (0:負, 1:正) }; struct BUNSU { struct NUMBER bunshi; // 分子 (BigInt) struct NUMBER bunbo; // 分母 (BigInt) int sign; };

アルゴリズムの実装

単純な加減乗除だけでなく、再帰的なロジックが必要なアルゴリズムも実装しています。 特に分数の約分処理（B_yakubun）においては、多倍長整数同士のユークリッドの互除法（gcd）を用いて最大公約数を求め、分母・分子を割り算する処理を実装しました。 これにより、計算結果が常に既約分数となるように制御しています。

エッジケースへの対応

引き算における繰り下がりや、負の数同士の掛け算など、符号処理でバグが発生しやすい箇所を重点的に検証しました。 絶対値の比較関数（numComp）を実装し、数値の大小関係に基づいて計算ロジックを分岐させています。

Reflections & Future / 振り返りと今後の課題

本ライブラリは学習初期に作成したものであり、現在は以下の改善点があることを認識しています。

  • メモリ効率：現在は1桁をint（32bit）で管理していますが、ビット演算を用いてuint64_t等に複数の桁を詰め込むことで、メモリ効率を改善できます。
  • 動的確保：桁数が固定長（#define KETA）ですが、mallocを用いた動的配列に変更することで、理論上メモリが許す限りの桁数を扱えるように改良可能です。
